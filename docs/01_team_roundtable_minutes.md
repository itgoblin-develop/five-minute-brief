## 2. 프론트엔드 리뷰

**🦊 Frontend Engineer**:
React 19 + Vite 7 + Tailwind CSS 4 조합은 최신 스택으로 잘 구성되어 있고, Radix UI 기반 컴포넌트도 접근성 측면에서 좋습니다. 그런데 가장 큰 문제는 **App.tsx가 "God Component"**라는 점입니다. 330줄에 20개 이상의 useState가 몰려 있고, 모든 콜백이 이 파일에서 시작해서 5단계 이상 prop drilling으로 내려갑니다. 새로운 기능 추가할 때마다 App.tsx를 건드려야 해서 병목이 됩니다.

**🎨 UX/UI Designer**:
SwipeDeck 카드 인터랙션은 예쁘지만, **비로그인 사용자가 3개 기사를 다 읽은 뒤의 경험이 너무 끊깁니다**. 로그인 모달만 띄우는데, 왜 로그인해야 하는지 가치를 전달하는 화면이 없어요. "무료로 모든 뉴스를 읽으세요" 같은 유도가 필요합니다. 또한 Pull-to-Refresh는 리스트 뷰에만 있고 카드 뷰에는 없는데, 사용자 입장에서 혼란스럽습니다.

**🦊 Frontend Engineer**:
맞아요. 그리고 **React Router 같은 라우팅 라이브러리가 없습니다.** history 배열로 자체 구현했는데, 브라우저 뒤로가기 버튼을 누르면 앱 자체를 벗어납니다. 딥링크도 안 되고, URL 공유도 불가능합니다.

**🐯 PO**:
모바일 웹이니까 URL 공유가 안 되면 바이럴 마케팅 자체가 불가능하다는 건데, 이건 꽤 치명적이네요.

**📢 Marketer**:
정확합니다. 뉴스 공유 기능이 핵심 성장 드라이버인데, 특정 기사에 딥링크가 없으면 카카오톡이나 SNS 공유가 의미가 없어집니다. 최소한 `/news/:id` 형태의 URL은 반드시 필요합니다.

**🦁 Lead Engineer**:
프론트엔드 쪽 정리하면 세 가지 구조적 이슈가 있는 겁니다. (1) App.tsx God Component 해체 → 상태관리 분리, (2) React Router 도입 → 딥링크/뒤로가기 지원, (3) 성능 최적화 → React.memo, useMemo, useCallback 적용. 이 세 가지만 해도 유지보수성과 사용자 경험이 크게 올라갑니다.

**🔍 QA**:
프론트엔드에 **테스트 코드가 하나도 없습니다.** 수동 테스트만으로는 회귀 버그를 잡을 수 없어요. 최소한 유틸리티 함수(getRelativeTime, formatCount)와 API 레이어에 대한 단위 테스트, 그리고 로그인 플로우에 대한 E2E 테스트는 있어야 합니다.

**🦊 Frontend Engineer**:
동의합니다. 그리고 Login.tsx, Signup.tsx 같은 **사용되지 않는 페이지 컴포넌트**가 남아 있어요. LoginModal로 대체됐는데 삭제하지 않았습니다.

## 3. 백엔드 리뷰

**🐻 Backend Engineer**:
Express 5 + PostgreSQL 조합은 안정적이고, parameterized query로 SQL Injection 방어도 잘 되어 있습니다. Helmet, CORS, rate-limit 등 보안 미들웨어도 잘 갖춰져 있고요. 문제는 크게 네 가지입니다.

첫째, **이메일 인증 코드가 인메모리 Map에 저장됩니다.** 서버 재시작하면 전부 날아가고, 수평 확장(multi-instance) 시 인증 코드가 공유되지 않습니다. 또한 만료된 코드를 정리하는 로직이 없어서 메모리 누수 위험이 있습니다.

**🐳 DevOps**:
이거 심각합니다. 프로덕션에서 PM2 cluster mode로 띄우거나 여러 인스턴스를 두면 바로 문제가 됩니다. Redis로 옮기는 게 필수입니다.

**🐻 Backend Engineer**:
둘째, **입력 검증이 수작업 regex입니다.** 이메일 regex가 4곳에 중복되어 있고, 비밀번호 regex도 3곳에 있습니다. Zod나 Joi 같은 스키마 검증 라이브러리를 도입하면 코드 중복과 에러 가능성을 동시에 줄일 수 있습니다.

셋째, **JWT 토큰 만료가 90일로 과도합니다.** Access Token은 15분~1시간, Refresh Token은 7~14일 정도가 표준입니다. 현재는 Access Token 하나로 90일을 쓰고 있어서 토큰 탈취 시 피해 범위가 큽니다.

**🦁 Lead Engineer**:
맞아요. Access Token / Refresh Token 이중 구조로 변경해야 합니다. 이미 `/api/auth/refresh` 엔드포인트는 있으니까 토큰 수명만 조정하면 됩니다.

**🐻 Backend Engineer**:
넷째, **비즈니스 로직이 라우트 핸들러에 직접 작성되어 있습니다.** auth.js가 531줄인데, 모든 로직이 라우트 콜백 안에 있습니다. Controller-Service-Repository 패턴으로 분리하면 테스트도 용이해지고 재사용성도 올라갑니다.

**🔍 QA**:
백엔드에도 **자동화 테스트가 전혀 없습니다.** API 엔드포인트만 32개 이상인데 수동으로 검증하는 건 한계가 있습니다. 최소한 인증 플로우와 CRUD 핵심 경로에 대한 통합 테스트가 필요합니다.

**🐻 Backend Engineer**:
추가로, **express.json()에 body size limit이 없습니다.** 악의적으로 거대한 JSON을 보내면 서버 메모리를 소진시킬 수 있어요. `{ limit: '1mb' }` 정도는 설정해야 합니다.

**🐳 DevOps**:
그리고 **Graceful Shutdown이 구현되어 있지 않습니다.** SIGTERM 시그널을 받았을 때 진행 중인 요청을 처리하고 DB 커넥션 풀을 정리한 뒤 종료해야 하는데, 현재는 그냥 죽습니다.

## 4. 파이프라인 리뷰

**🦁 Lead Engineer**:
AI 재구성 파이프라인은 아키텍처적으로 상당히 잘 설계되어 있습니다. 전처리 → 클러스터링 → LLM 재구성 → 검증 → DB 적재의 5단계가 각각 독립 모듈로 분리되어 있고, LLM Router의 Primary/Fallback 자동 전환은 안정성 면에서 훌륭합니다.

**🐻 Backend Engineer**:
그런데 **Selenium 기반 크롤링이 불안정합니다.** 네이버 뉴스 UI가 바뀌면 CSS Selector가 깨지고, YouTube도 마찬가지입니다. 크롤링 실패 시 알림이 GitHub Issue 생성밖에 없는데, Slack 웹훅 같은 즉각적인 알림이 필요합니다.

**🐳 DevOps**:
GitHub Actions에서 Selenium + Chrome을 돌리는 건 비용 대비 효율이 떨어집니다. 셀레니움 대신 Naver는 RSS나 API를 활용하고, YouTube는 이미 API v3 크롤러가 별도로 있으니 그걸 메인으로 쓰는 게 낫습니다.

**🦁 Lead Engineer**:
동의합니다. 그리고 `run_daily.py`에서 Step 1(크롤링)의 타임아웃이 600초, Step 2(재구성)가 900초인데, **타임아웃 시 에러 핸들링이 subprocess 레벨**입니다. 개별 단계의 실패를 더 세밀하게 처리하고, 부분적 성공도 허용하는 로직이 필요합니다.

**🔍 QA**:
파이프라인 산출물의 **품질 검증 결과가 로그로만 남습니다.** 검증 통과율, 유사도 경고 비율 같은 메트릭을 DB에 저장하거나 대시보드로 시각화하면 일별 품질 추이를 모니터링할 수 있습니다.

**🐯 PO**:
파이프라인이 매일 07:00 KST에 도는데, **파이프라인 실패 시 사용자에게 빈 피드가 보입니다.** 전날 데이터를 폴백으로 보여주는 로직이 백엔드에 있나요?

**🐻 Backend Engineer**:
없습니다. 현재 `/api/news` 엔드포인트는 `published_at DESC`로 정렬해서 반환하기 때문에, 오늘 데이터가 없으면 어제 데이터가 보이긴 합니다. 하지만 "오늘의 브리핑"이라는 컨셉과는 맞지 않아요.

## 5. UX/디자인 리뷰

**🎨 UX/UI Designer**:
전체적인 디자인 퀄리티는 높습니다. Primary Blue (#3D61F1), 카테고리 컬러, SwipeDeck 인터랙션 등이 잘 어우러져 있어요. 몇 가지 개선점을 짚겠습니다.

1. **온보딩 플로우가 없습니다.** 첫 방문 사용자가 앱에 들어오면 바로 뉴스 피드가 보이는데, 서비스 가치 전달 없이 시작합니다. 간단한 3~4단계 온보딩(카테고리 선택, 알림 시간 설정)이 필요합니다.

2. **댓글 삭제가 실제로 동작하지 않습니다.** NewsDetail 컴포넌트에서 삭제 버튼 클릭 시 toast만 띄우고 실제 API 호출이 없습니다.

**🔍 QA**:
잠깐, 이건 버그인가요?

**🦊 Frontend Engineer**:
맞습니다. 댓글 삭제 버튼의 onClick에서 `toast.success("댓글이 삭제되었습니다.")`만 호출하고 `interactionAPI.deleteComment()`를 호출하지 않습니다. 수정 기능도 `toast.info("수정 기능 준비 중입니다.")`로 되어 있어서 사실상 미구현입니다.

**🔍 QA**:
이건 심각한 기능 누락입니다. 사용자 입장에서 삭제가 성공했다고 메시지가 뜨는데 실제로는 안 지워진 거면 신뢰 문제가 됩니다.

**🎨 UX/UI Designer**:
3. **다크 모드 미지원.** 출근길 새벽이나 밤에 쓸 수 있는 서비스인데 다크 모드가 없으면 눈이 피로합니다. 다만 이건 MVP 이후로 미뤄도 됩니다.

4. **알림(Notifications) 페이지의 데이터가 하드코딩되어 있습니다.** 실제 알림 시스템이 없고 더미 데이터만 보여줍니다.

**📢 Marketer**:
온보딩과 관련해서, **관심 카테고리 선택 기능**이 있으면 초기 리텐션을 크게 올릴 수 있습니다. 사용자가 "경제"와 "트렌드"만 선택하면 해당 카테고리만 우선 노출하는 거죠.

## 6. 보안 및 운영 리뷰

**🐳 DevOps**:
보안 관점에서 정리합니다.

1. **JWT 90일 만료**: 위에서 언급한 대로 과도합니다.
2. **Rate Limiting이 auth에만 적용**: 댓글 스팸, 좋아요 어뷰징 방어가 없습니다.
3. **계정 열거(Account Enumeration)**: 로그인 실패 시 "이메일이 존재하지 않습니다" vs "비밀번호가 틀렸습니다"를 구분해서 알려줍니다. 공격자가 이메일 존재 여부를 확인할 수 있습니다.
4. **DB 마이그레이션 도구 없음**: `setup_database.sql` 하나로 관리하고 있어서 스키마 변경 이력 추적이 불가능합니다.
5. **프로덕션 배포 설정 없음**: Docker Compose, Nginx reverse proxy, SSL/TLS 등이 아직 구성되지 않았습니다.

**🐻 Backend Engineer**:
추가로, **news 테이블에 pagination이 OFFSET 기반**입니다. 데이터가 쌓이면 OFFSET이 커질수록 성능이 떨어집니다. cursor-based pagination으로 전환해야 합니다.

**🦁 Lead Engineer**:
현재 단계에서는 데이터 양이 적어서 큰 문제는 아니지만, 장기적으로는 맞습니다. 다만 우선순위는 낮을 수 있어요.

## 7. 마케팅/비즈니스 관점

**📢 Marketer**:
비즈니스 관점에서 몇 가지 짚겠습니다.

1. **프리미엄(유료) 모델 전환 기반이 없습니다.** 현재 비로그인 3개 제한인데, 로그인하면 무제한입니다. 유료 전환 유도 포인트가 없어요.

2. **공유 기능이 미약합니다.** 기사 공유 버튼은 있지만 딥링크가 없어서 실효성이 없습니다.

3. **SEO 최적화가 전혀 되어 있지 않습니다.** SPA(Single Page Application)이라 검색 엔진 크롤러가 콘텐츠를 읽을 수 없습니다. SSR이나 최소한 메타 태그 동적 생성이 필요합니다.

4. **사용자 분석 도구(GA, Amplitude 등)가 없습니다.** 어떤 카테고리가 인기 있는지, 평균 체류 시간이 얼마인지 데이터 기반 의사결정이 불가합니다.

**🐯 PO**:
SEO는 현재 모바일 웹이니까 장기적으로 중요한데, MVP에서는 후순위로 두겠습니다. 하지만 GA 연동은 즉시 해야 합니다. 사용자 행동 데이터 없이 서비스 개선 방향을 잡을 수 없으니까요.

## 8. 핵심 논쟁 사항

### 논쟁 1: React Router 도입 시점

**🦊 Frontend Engineer**: 지금 당장 도입해야 합니다. 딥링크, 뒤로가기, URL 공유 모두 필수이고 늦추면 마이그레이션 비용만 커집니다.

**🐯 PO**: 배포 일정이 촉박한데, React Router 마이그레이션이 며칠이나 걸리나요?

**🦊 Frontend Engineer**: 현재 뷰가 7~8개이고 구조가 단순하니까, 2~3일이면 충분합니다. 하지만 그동안 다른 프론트 작업이 멈춥니다.

**🎨 UX/UI Designer**: 딥링크 없이는 공유 기능이 의미가 없으니 배포 전에 반드시 해야 합니다.

**결론**: **배포 전 필수 작업으로 확정.**

### 논쟁 2: 상태관리 라이브러리 도입

**🦊 Frontend Engineer**: Zustand를 도입하면 App.tsx 리팩토링이 깔끔해집니다.

**🦁 Lead Engineer**: Context API만으로도 충분하지 않나요? 라이브러리를 하나 더 추가하는 것보다 Context를 잘 나누는 게 나을 수 있습니다.

**🦊 Frontend Engineer**: Context는 Provider가 많아지면 wrapper hell이 됩니다. Zustand는 보일러플레이트가 거의 없고 번들 사이즈도 1KB입니다.

**결론**: **Zustand 도입에 합의. 단, 점진적 마이그레이션.**

### 논쟁 3: Redis 도입 범위

**🐻 Backend Engineer**: 인증 코드만 Redis로 옮기면 됩니다.

**🐳 DevOps**: 어차피 Redis 인스턴스를 띄우는 거라면, 뉴스 목록 캐싱도 같이 하는 게 효율적입니다. 뉴스 데이터는 하루에 한 번만 갱신되니까 TTL 24시간으로 캐싱하면 DB 부하를 크게 줄일 수 있습니다.

**🐻 Backend Engineer**: 동의합니다. 다만 캐시 무효화 로직도 같이 짜야 해서 시간이 좀 더 걸립니다.

**결론**: **Phase 1에서 인증 코드 Redis 이전, Phase 2에서 뉴스 캐싱 추가.**
